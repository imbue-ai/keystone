# Pitfalls Research

**Domain:** Modal sandbox observability ‚Äî diagnosing intermittent cloud execution failures
**Researched:** 2026-03-09
**Confidence:** HIGH (codebase inspection + Modal official docs) / MEDIUM (Modal-specific failure modes)

---

## Critical Pitfalls

### Pitfall 1: Treating `except Exception: pass` as "safe cleanup"

**What goes wrong:**
`ManagedProcess._stream_reader` (modal_runner.py:91-93) has a bare `except Exception: pass` that silently consumes whatever exception the Modal SDK raises when a sandbox terminates unexpectedly. When the sandbox dies mid-stream, the thread exits cleanly with no log entry. The parent sees "stream finished" and can't distinguish between a clean agent exit and a killed sandbox. This is the primary reason the current 45-minute failures are invisible.

**Why it happens:**
Stream readers are daemon threads; if the process ends normally, the stream iteration completes without exception. The `except Exception: pass` was added to handle the expected "stream closed" case on sandbox cleanup. But it also swallows the unexpected case ‚Äî a killed sandbox and a normally-terminated one look identical to the caller.

**How to avoid:**
Replace `pass` with a logged warning that includes the exception type, message, and stream name. Do not re-raise ‚Äî stream termination is not recoverable at this level. The log entry is what matters.

```python
except Exception as exc:
    logger.warning(
        "[%s] %s stream closed unexpectedly: %s: %s",
        self.prefix, stream_name.value, type(exc).__name__, exc,
    )
```

The Modal SDK raises `SandboxTerminatedError` (internal reason) or `SandboxTimeoutError` (duration exceeded) or `ResourceExhaustedError` (quota/rate limit) ‚Äî all distinct types that appear in the log and immediately narrow the investigation.

**Warning signs:**
- Failures produce no exception or traceback in logs whatsoever
- "Stream finished" appears in logs but no subsequent "agent exit code" log appears
- The only symptom is the eval harness timing out waiting for results

**Phase to address:** Phase 1 (fix `_stream_reader`). This is the most immediate change with the highest diagnostic return. Do it first, before even threading the sandbox ID, because it produces value on the next eval run regardless of whether sandbox IDs are persisted.

### Pitfall 2: Losing the sandbox ID before the failure can be investigated

**What goes wrong:**
`ModalAgentRunner.ensure_sandbox()` captures `self._sandbox.object_id` and prints it to stderr (modal_runner.py:183-184), but never returns it to any caller or stores it in a retrievable structure. When the sandbox dies, the stderr print is the only record ‚Äî and in the Prefect parallel eval harness, stderr from a subprocess is either truncated to 1000 bytes (evals/flow.py:380) or not preserved at all. The sandbox ID is gone before anyone can use it to query the Modal dashboard or API.

**Why it happens:**
The sandbox ID was added for interactive debugging ("copy the ID and run `modal shell sb-xxx`"), not for automated post-mortem. In single-repo runs this is fine. In parallel evals, the subprocess stderr is captured and may be truncated; even if not, mapping a stderr line to a specific failed eval repo requires manual correlation.

**How to avoid:**
Thread `object_id` into `BootstrapResult` via a `get_sandbox_id()` getter method on `AgentRunner`. `BootstrapResult` is serialized to JSON and stored in `cli_run.bootstrap_result_json` and written to `keystone_result.json` ‚Äî both of which survive sandbox death and are available for post-mortem queries. See ARCHITECTURE.md for the complete data-flow design.

**Warning signs:**
- Post-mortem investigation starts with "which sandbox was it?" and there is no answer
- Eval result JSON contains no sandbox identifier
- The Modal dashboard shows terminated sandboxes but there is no way to match them to specific eval repos

**Phase to address:** Phase 2 (thread sandbox ID to `BootstrapResult`). Requires Phase 1 to be in place first so the stream reader logs give context alongside the persisted ID.

### Pitfall 3: Adding sandbox ID to the cache key or `AgentConfig`

**What goes wrong:**
If `modal_sandbox_id` is added to `AgentConfig`, `CacheKey`, or `AgentRunRecord`, every new eval run (which creates a new sandbox with a new ID) becomes a cache miss. The caching layer, which deduplates expensive agent runs, is silently destroyed. This is subtle because the code still runs correctly ‚Äî it just runs every agent from scratch every time.

**Why it happens:**
Sandbox ID looks like "configuration" of the run, so it seems natural to put it alongside other run parameters. But it is an output (generated by Modal when the sandbox is created), not an input that determines what the agent does.

**How to avoid:**
The sandbox ID belongs only in `BootstrapResult` (the final output record), accessible via `get_sandbox_id()` called after `run()` returns. It must never appear in `CacheKey`, `AgentConfig`, `AgentRunRecord`, or any field that feeds the cache key computation. See ARCHITECTURE.md anti-pattern section for the full rationale.

**Warning signs:**
- Cache hit rate drops to 0% after the change
- Every eval task shows "cache miss" even for repos run previously with identical configs
- `agent_run` table shows duplicate entries for the same `git_tree_hash` + `prompt_hash` + `agent_config`

**Phase to address:** Phase 2 design checkpoint. Verify the cache key computation does not touch sandbox ID before merging.

### Pitfall 4: Building the post-mortem script before sandbox IDs exist in the DB

**What goes wrong:**
The post-mortem script (Phase 3) queries `cli_run.bootstrap_result_json` for `modal_sandbox_id` values, then calls the Modal SDK to retrieve sandbox history. If Phase 2 has not shipped yet and been run against production eval jobs, every `json_extract(..., '$.modal_sandbox_id')` returns NULL. The script appears to work (no errors) but returns no results. Time is wasted debugging the script when the real problem is the absence of data.

**Why it happens:**
It is tempting to write the post-mortem tooling in parallel with the data-threading changes, to be "ready to use" as soon as IDs start appearing. But until a real eval run completes with Phase 2 deployed, there is nothing to query.

**How to avoid:**
Build Phase 3 after at least one production eval run has completed with Phase 2 deployed. Verify manually that `json_extract(bootstrap_result_json, '$.modal_sandbox_id')` returns non-NULL values before spending time on the script. The script itself is simple; the value is in having IDs to work with.

**Warning signs:**
- Script runs without errors but returns 0 rows or all-NULL sandbox IDs
- No failures have occurred since Phase 2 deployed (which means there is nothing to investigate yet anyway)

**Phase to address:** Phase 3 (post-mortem script). Sequence dependency: Phase 2 must ship and be exercised first.

### Pitfall 5: Assuming the 45-minute failure is a timeout misconfiguration

**What goes wrong:**
The sandbox is created with `timeout=self._timeout_seconds * 2` (modal_runner.py:178), which is 4 hours for a 2-hour `time_limit_seconds` config. The outer timeout is deliberately generous. Assuming the fix is adjusting a timeout value leads to chasing the wrong root cause. The more likely explanations are:

- **Modal-side OOM kill**: The sandbox runs a Docker daemon plus the agent's Docker build. Docker-in-Docker (DinD) doubles the memory pressure ‚Äî the outer Modal container runs a daemon, and that daemon runs another container for verification. Modal's default container allocation is 128 MiB RAM; even the configured reservation may be too small under large parallel load where the host is oversubscribed.
- **Modal `ResourceExhaustedError`**: Under large parallel loads, Modal raises `ResourceExhaustedError` when account-level quotas are hit. This is an SDK exception, not a timeout. Without logging in `_stream_reader`, it is invisible.
- **`idle_timeout` being set unintentionally**: Modal added an `idle_timeout` parameter to `Sandbox.create()` (SDK v1.3+). If this is set somewhere in the config chain, a sandbox with no active `sb.exec()` call (e.g. during a pause between agent execution and verification) could be terminated.
- **gRPC connection failure between Prefect worker and Modal**: Long-lived connections from Prefect workers to Modal's gRPC API can be reset by intermediate infrastructure after ~45 minutes of low traffic, causing the streaming iterator to raise a connection error.

**How to avoid:**
Log the specific exception type from `_stream_reader`. After one logging-enabled eval run, the exception type will identify the root cause: `SandboxTerminatedError` (Modal killed it), `ResourceExhaustedError` (quota), `ConnectionError` (network), or something else entirely. Do not change timeouts, quotas, or memory settings before you know which exception is being raised.

**Warning signs:**
- Failures cluster at roughly the same wall-clock time (45 min) regardless of repo complexity ‚Äî suggests an infrastructure-level trigger rather than agent workload
- Failures only occur when N sandboxes are running simultaneously ‚Äî consistent with quota exhaustion or host oversubscription
- Failures never occur during single-repo test runs ‚Äî rules out a code bug in the agent path itself

**Phase to address:** Phase 1 (logging) is the diagnostic tool. The actual cause cannot be known until Phase 1 data exists.

### Pitfall 6: Misidentifying which layer the failure occurs at

**What goes wrong:**
The failure could be at any of four layers: (1) the agent process inside the sandbox OOM-kills, (2) the Docker daemon inside the sandbox terminates, (3) the Modal sandbox container itself is killed by Modal's infrastructure, or (4) the gRPC stream between the Prefect worker and Modal's API is reset. Each layer has different signals and different fixes. Without distinguishing them, remediation is guesswork.

**How to avoid:**
The logged exception type from `_stream_reader` (Pitfall 1 fix) narrows to layer 3 or 4. For layers 1 and 2 (agent/Docker within the sandbox), the signals would appear in normal log output before the stream terminates ‚Äî not as stream exceptions. If the exception is `SandboxTerminatedError`, it is layer 3 (Modal killed the container). If it is a gRPC/connection error (`ConnectionError`, `grpc.RpcError`), it is layer 4. If the stream closes cleanly (no exception) but the agent produces no output, layers 1 or 2 are suspect.

**Warning signs:**
- Clean stream termination with no exception but also no agent exit code: suspect layer 1/2
- `SandboxTerminatedError` in logs: Modal killed the container (layer 3) ‚Äî check OOM or quota
- `ResourceExhaustedError`: quota/rate limit (layer 3 variant)
- Any gRPC error or `ConnectionError`: network layer (layer 4) ‚Äî consider reconnection or timeout tuning

**Phase to address:** Phase 1 (logging). The exception type from the first post-Phase-1 failing run will resolve this.

## Technical Debt Patterns

Shortcuts relevant to this milestone. These exist in the codebase today and interact with the observability work.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Bare `except Exception: pass` in `_stream_reader` | Avoids noisy logs from expected stream-close on cleanup | Hides all unexpected termination events, making failures invisible | Never ‚Äî replace with logged warning before this milestone ships |
| Printing sandbox ID to stderr instead of persisting it | Quick to implement, visible in interactive use | Lost in automated eval runs; unavailable for post-mortem | Never for production eval runs ‚Äî was a prototype behavior |
| stderr truncation to first 1000 bytes in `evals/flow.py:380` | Limits log volume | Cuts off the actual error that usually appears at the end | Acceptable for non-critical logs; wrong for sandbox failure context |
| No exception type specificity in Modal calls | Simpler code | Cannot distinguish OOM from timeout from quota from network reset | Acceptable only in throw-away prototypes |

## Integration Gotchas

Specific to the Modal SDK and how it integrates with the Prefect-based eval harness.

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Modal SDK exceptions | Catching `Exception` broadly ‚Äî masks `SandboxTerminatedError` vs `ResourceExhaustedError` vs `ConnectionError` | Catch specific Modal exception types; log the type name always |
| `modal container logs <sandbox-id>` CLI | Assuming this works for any sandbox ID at any time post-termination | Log availability window after termination is not documented; use it promptly; SDK `from_id()` may be more reliable for scripted access |
| `modal.Sandbox.terminate()` vs `detach()` | Calling `terminate()` and assuming the sandbox is done ‚Äî since SDK v1.0.3 it returns before shutdown completes | Call `sb.wait(raise_on_termination=False)` after `terminate()` if you need to confirm shutdown, or use `terminate(wait=True)` (v1.3.4+) |
| `idle_timeout` on `Sandbox.create()` | Not setting it deliberately but a future SDK default or config file silently enables it | Explicitly verify `idle_timeout` is not set in `Sandbox.create()` call; check Modal SDK changelog on upgrades |
| Docker-in-Docker memory | Assuming Modal container memory limit covers only the outer container | Docker daemon + inner containers consume memory within the same Modal container memory limit. 128 MiB default is insufficient; if not already set, request explicit memory via `cpu` and `memory` parameters |
| Prefect task subprocess stderr | Assuming subprocess stderr is fully captured for post-mortem | `evals/flow.py:380` truncates to first 1000 bytes. Sandbox ID must be in `keystone_result.json` (Step 2 target), not relied upon from stderr capture |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Logging every stream chunk at DEBUG level | Log volume grows linearly with agent output; disk fills under parallel eval loads | Log at INFO for important events, DEBUG for raw stream data; use log level filtering | At ~50+ parallel sandboxes with verbose agents |
| `modal container logs` in a loop for many sandboxes | CLI invocations are slow; serial loop over 100 sandbox IDs takes minutes | Batch or parallelize using `modal.Sandbox.from_id()` in the SDK rather than CLI | Immediately for any post-mortem loop over more than ~10 sandboxes |
| Post-mortem script using `json_extract` on every row without an index | Full table scan on `cli_run`; slow as table grows | Add `WHERE created_at > ?` filter to limit to recent runs; SQLite JSON path queries are unindexed | After a few hundred thousand rows |

## "Looks Done But Isn't" Checklist

Items that are easy to miss when the implementation looks correct at first glance.

- [ ] **`_stream_reader` logging**: Exception is logged ‚Äî but verify the log message includes `type(exc).__name__` not just `str(exc)`. Some Modal exceptions have empty string representations; the type name is the diagnostic signal.
- [ ] **Sandbox ID in `BootstrapResult`**: Field is added and populated ‚Äî but verify it also appears in `keystone_result.json` written by the CLI (not just in the DB), because the eval harness reads that file, not the DB.
- [ ] **Cache hit path**: `CachedAgentRunner.get_sandbox_id()` returns `None` on cache hits ‚Äî confirm this is not treated as an error by `keystone_cli.py`. `None` is the correct value for a cached replay.
- [ ] **Backward compatibility**: Old `BootstrapResult` JSON rows (without `modal_sandbox_id`) still deserialize correctly via Pydantic's optional-field default. Test by loading an old row from the DB.
- [ ] **Post-mortem script scope**: Script correctly filters to the time window of interest (e.g. last N runs, or runs from a specific eval config name). A script that dumps all 10,000 historical sandbox IDs is not useful for diagnosing a specific failure event.
- [ ] **Modal SDK version check**: `modal container logs <sandbox-id>` support was added in a specific SDK version. Verify the installed version supports it before documenting it as the post-mortem tool.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Phase 1 shipped without sandbox ID (can't correlate logs to repos) | MEDIUM | Cross-reference timestamps: use the log timestamps from `_stream_reader` warnings vs. eval task start/end times in Prefect. Imprecise but workable for large time spreads. |
| Phase 2 shipped with sandbox ID in cache key | HIGH | Must roll back the cache key change and invalidate all cache entries (or change `cache_version` string). All cached agent runs become re-runs. |
| Post-mortem script built before IDs exist in DB | LOW | Wait for next failing eval run after Phase 2 ships. No code change needed; data will appear automatically. |
| `idle_timeout` triggered the failures | LOW | Remove or raise `idle_timeout` in `Sandbox.create()` call. The current call in `ensure_sandbox()` (modal_runner.py:175-181) does not set `idle_timeout`, so this is only a risk if a future SDK default introduces it. |
| OOM kill confirmed as root cause | MEDIUM | Increase Modal container memory request via `memory` parameter in `Sandbox.create()`. Requires knowing how much Docker-in-Docker actually needs; instrument with `docker stats` output captured via `run_modal_command`. |
